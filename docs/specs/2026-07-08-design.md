# CI/テスト・オブザーバビリティツールtestlake設計ドキュメント

日付: 2026-07-08
ステータス: 設計承認待ち

## 1. 概要

GitHub Actions のテスト結果・ビルドメタデータを Parquet として蓄積し、DuckDB-Wasm の静的ビューアで分析する CI オブザーバビリティツール。サーバーを1台も立てず、公開リポジトリなら追加費用ゼロで動く。

**一言で**: 「Develocity の Failure Analytics / Build Analytics 相当を、GitHub Pages で動く OSS に」

## 2. 背景と狙い

- CI 分析(flaky テスト検出、テスト時間トレンド、ビルド時間リグレッション)の領域は、Develocity(エンタープライズ価格)と SaaS(BuildPulse、Trunk 等)しかなく、セルフホスト OSS の定番が存在しない空白地帯
- 検証済みの技術的裏付け: CI テレメトリは1日あたり数 MB〜数十 MB 規模であり、シングルスレッドの DuckDB-Wasm がブラウザ内で快適に処理できる。「取込 → Parquet → 静的ビューア」というゼロサーバー構成が、o11y 全域(ログ・トレース・メトリクス)では性能的に破綻するのに対し、CI データでは無理なく成立する
- ターゲット: スタートアップ・小規模チーム、および OSS プロジェクト。「flaky テストの再実行ばかりしている」「CI が遅くなった原因が分からない」という普遍的なペインに、workflow へ1ステップ追加するだけで応える

## 3. スコープ

### v0.1(MVP)

1. **収集 Action**(`kyosu-1/testlake`): JUnit XML のパース + GitHub Actions コンテキストの収集 → Parquet 化 → `gh-pages` ブランチへプッシュ
2. **静的ビューア**(DuckDB-Wasm SPA): flaky ランキング / テスト時間トレンド / 失敗率 / ビルド時間推移 / SQL コンソール
3. **ストレージレイアウト規約**: パーティション構成とスキーマの公開ドキュメント

### v0.2 以降(明示的にMVP外)

- PR 自動コメント(「このPRで遅くなったテスト Top5」「新規 flaky 化」)
- リグレッション閾値超過による CI fail ゲート
- プライベートリポジトリ向け R2/S3 バックエンドと認証付きビューア
- ホスティング版ビューア(`viewer.testlake.dev?src=...`)
- 失敗メッセージのグルーピング分析

### 非スコープ(やらないこと)

- GitHub Actions 以外の CI 対応(CircleCI/GitLab/Jenkins)。導入体験の磨き込みを優先し、意図的に特化する
- 実行時テレメトリ(アプリのログ・トレース・メトリクス)。本ツールは CI データ専用
- サーバーサイドのクエリ実行系。いかなる形でも常駐プロセスを持たない

## 4. アーキテクチャ

```
GitHub Actions ジョブ
  └─ kyosu-1/testlake@v1 (composite action + Go 単一バイナリ)
       ├─ JUnit XML パース (test-results/**/*.xml)
       ├─ Actions コンテキスト収集 (GITHUB_SHA, run_id, attempt, workflow, job, runner, 所要時間)
       ├─ Parquet 書き出し
       └─ gh-pages ブランチ ci-data/ へ commit & push (競合時は rebase リトライ)

GitHub Pages (静的配信, CORS 可)
  ├─ ci-data/            … Parquet データ + manifest.json
  └─ ci/ (任意)          … ビューア SPA 一式(Action が viewer: true で同梱デプロイ)

ブラウザ (DuckDB-Wasm)
  └─ manifest.json を読み、必要な Parquet だけ HTTP Range で取得して描画
```

構成要素は「Action(書き手)」「レイアウト規約(契約)」「ビューア(読み手)」の3ユニット。書き手と読み手は Parquet スキーマという契約だけで疎結合であり、それぞれ独立にテスト・差し替え可能。

## 5. コンポーネント設計

### 5.1 収集 Action(`kyosu-1/testlake`)

- 実体は Go の単一バイナリを composite action でラップしたもの
- 入力(with):
  - `reports`: JUnit XML の glob(必須)
  - `viewer`: `true` でビューア SPA を gh-pages に同梱デプロイ(デフォルト false)
  - `branch`: データ書き込み先ブランチ(デフォルト `gh-pages`)
- `if: always()` での利用を前提とし、テスト失敗時もデータを収集する
- 書き込みは「gh-pages を shallow fetch → Parquet 追加 → commit → push、失敗したら rebase して最大5回リトライ」。並行するジョブが同時にプッシュしても、ファイル名が run_id + job + attempt で一意なため rebase は常に成功する
- 権限は標準の `GITHUB_TOKEN`(`contents: write`)のみ。外部サービスへの通信は一切しない

### 5.2 ストレージレイアウト規約

```
ci-data/
  manifest.json                          # ファイル一覧・スキーマバージョン・統計(ビューアの索引)
  runs/date=YYYY-MM-DD/<run_id>-<job>-<attempt>.parquet
  tests/date=YYYY-MM-DD/<run_id>-<job>-<attempt>.parquet
```

- **runs テーブル**(ジョブ実行ごとに1行): run_id, run_attempt, workflow, job, branch, sha, event, runner_os, started_at, duration_ms, conclusion
- **tests テーブル**(テストケース実行ごとに1行): run_id, run_attempt, job, suite, class, name, file, outcome(passed/failed/skipped/error), duration_ms, failure_message(先頭4KBに切り詰め), failure_type
- manifest.json は Action が毎回再生成。ビューアはこれだけ読めば GitHub API を呼ばずにファイル一覧と日付範囲を把握できる
- スキーマにはバージョン番号を持たせ、後方互換の破壊はメジャーバージョンでのみ行う
- **コンパクション**: 小ファイル増殖対策として、Action が「7日より古い日付ディレクトリの複数 Parquet を日次1ファイルにマージ」する処理を実行のついでに行う(獲得ロックは不要 — 対象が古い日付のため並行書き込みと衝突しない)

### 5.3 静的ビューア(DuckDB-Wasm SPA)

- ビルド成果物はただの静的ファイル一式。`?src=<ci-dataのURL>` でデータ位置を指定(同梱デプロイ時は相対パスで自動解決)
- 画面構成:
  1. **Flaky ランキング**: flaky 判定は「同一 sha で成功と失敗の両方を記録した(リトライ含む)テスト」。スコアは直近30日の flake 回数を日次減衰で重み付け。各テストの直近50実行の成否タイムラインを表示
  2. **テスト時間トレンド**: スイート/ファイル/テスト単位の duration 推移(p50)。期間比較で悪化順ソート
  3. **失敗分析**: ブランチ別・ワークフロー別の失敗率、失敗回数上位テスト
  4. **ビルド時間**: workflow/job 別 duration 推移。直近7日 vs 先行30日の p50 比較で有意な悪化をハイライト
  5. **SQL コンソール**: `runs` / `tests` ビューを定義済みの DuckDB-Wasm シェル。任意クエリを実行可能
- すべての集計は事前に用意した SQL(DuckDB 方言)で実装し、アプリケーションロジック側での集計は行わない。ダッシュボードの各パネル = 1 SQL、という対応を保つ

## 6. データフローとエラー処理

- **不正な JUnit XML**: 寛容にパースし、読めたテストケースだけ記録。パース失敗はジョブを fail させず warning 出力(収集ツールが CI を壊すことは絶対にしない)
- **push 競合**: rebase リトライ(最大5回)。それでも失敗した場合はデータを Actions Artifact として退避し warning
- **巨大な失敗メッセージ**: 4KB で切り詰め(スタックトレース対策)
- **データ肥大**: コンパクション + `retention_days`(デフォルト 400 日)超過分の削除
- **タイムゾーン**: すべて UTC で記録、表示時にローカル変換
- **ビューアのデータ欠損**: manifest に存在するがフェッチできないファイルはスキップして描画継続、欠損バナーを表示

## 7. セキュリティ・プライバシー

- v0.1 は公開リポジトリ前提(gh-pages = 公開)。README に「テスト名・失敗メッセージが公開されること」を明記
- Action は `GITHUB_TOKEN` 以外の資格情報を扱わない。テレメトリの外部送信なし
- プライベートリポジトリ対応(R2 + 認証)は v0.2 で設計する

## 8. テスト戦略

- **パーサ単体テスト**: pytest / Jest / Go test / Gradle / Maven の実 JUnit XML サンプルをフィクスチャ化し、ゴールデンテスト
- **Action 統合テスト**: テスト用リポジトリで実ワークフローを回し、gh-pages に期待どおりの Parquet + manifest が生成されることを検証。並行ジョブでの push 競合シナリオを含む
- **ビューアテスト**: 生成済みフィクスチャ Parquet に対し、各パネルの SQL が期待値を返すことを DuckDB(ネイティブ)で検証。SQL はネイティブ/Wasm で共通なので、ブラウザ E2E は smoke レベルに留める
- **ドッグフーディング**: 本リポジトリ自身の CI に v0.1 を組み込み、自分のダッシュボードを公開する(これ自体がデモになる)

## 9. 主要リスクと対応

| リスク | 対応 |
|---|---|
| gh-pages 運用がユーザーの既存 Pages 利用と衝突 | データはサブディレクトリ `ci-data/` に閉じる。既存 Pages コンテンツには触れない |
| GitHub リポジトリ/Pages のサイズ制限(~1GB 推奨) | コンパクション + retention で通常利用なら数年分収まる。超大規模チームは v0.2 の R2 バックエンドへ誘導 |
| raw/Pages の CORS・キャッシュ挙動 | GitHub Pages は `Access-Control-Allow-Origin: *` を返す。キャッシュ(10分)による鮮度遅延は許容範囲として明記 |
| DuckDB-Wasm の初期ロード(~数MB) | ビューアは分析ツールであり毎秒開くものではない。Wasm は CDN キャッシュ + プリフェッチで緩和 |
